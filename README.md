# プロジェクト概要

## 目的

本リポジトリは、プロジェクトの要件定義・設計資料を体系的に管理するためのドキュメントリポジトリです。

- **NotebookLMで読みやすく**: 1ファイル=1トピックの構成で、AIツールでの解析・要約を容易にする
- **Cursor/開発者が参照しやすい**: 明確なディレクトリ構造と相互リンクで、必要な情報に素早くアクセス
- **将来の拡張や差分管理を容易に**: ファイル単位での変更履歴管理と、明確な命名規則

## 想定ユーザー

- プロジェクトマネージャー
- エンジニア（フロントエンド・バックエンド・インフラ）
- デザイナー
- プロダクトオーナー
- ステークホルダー

## スコープ

### 含むもの

- 機能要件（Functional Requirements）
- 非機能要件（Non-Functional Requirements）
- システム設計（アーキテクチャ・API・データモデル）
- UI/UX設計（ワイヤーフレーム・画面遷移・コンポーネント）
- プロジェクト管理資料（ロードマップ・WBS・リスク・意思決定）
- 用語集・参考資料・変更履歴

### 含まないもの

- 実装コード（別リポジトリで管理）
- テストコード・テスト仕様書（別リポジトリまたは別ディレクトリで管理）
- 運用マニュアル（別リポジトリで管理）

## ディレクトリ構成

```
project-root/
├─ 0_Overview/          # プロジェクト概要・ビジョン・ステークホルダー
├─ 1_FunctionalRequirements/  # 機能要件（FR_xxx_*.md）
├─ 2_NonFunctionalRequirements/  # 非機能要件
├─ 3_SystemDesign/      # システム設計（アーキテクチャ・API・データモデル）
├─ 4_UIUX/             # UI/UX設計
├─ 5_ProjectManagement/  # プロジェクト管理（ロードマップ・WBS・リスク・意思決定）
├─ 6_Appendix/         # 付録（用語集・参考資料・変更履歴）
├─ SUMMARY.md          # 全ドキュメントへの目次・リンク集
└─ README.md           # 本ファイル
```

詳細は [SUMMARY.md](./SUMMARY.md) を参照してください。

## ドキュメント運用ルール

### 命名規則

- **機能要件ファイル**: `FR_XXX_機能名.md`（例: `FR_001_user_authentication.md`）
- **非機能要件ファイル**: `小文字_アンダースコア区切り.md`（例: `performance_and_scalability.md`）
- **設計ファイル**: `小文字_アンダースコア区切り.md`（例: `architecture_overview.md`）
- **管理ファイル**: `小文字_アンダースコア区切り.md`（例: `risk_log.md`）

### ファイル構造

すべてのMarkdownファイルは以下の構造に従います：

1. **YAML Front Matter**: タイトル・更新日・オーナー
2. **本文**: セクションごとに整理
3. **TODOセクション**: ファイル末尾に初期タスクを記載

### PR運用

- **ブランチ命名**: `docs/update-<ファイル名>` または `docs/add-<機能名>`
- **コミットメッセージ**: `docs: <変更内容>` の形式
- **PRチェックリスト**:
  - [ ] YAML Front Matterが更新されているか
  - [ ] 関連ドキュメントへの相互リンクが追加されているか
  - [ ] SUMMARY.mdが更新されているか（新規ファイル追加時）
  - [ ] 変更履歴（change_history.md）が更新されているか
  - [ ] レビュー依頼先が指定されているか

### 更新頻度

- **機能要件・設計**: 要件変更・設計変更時に即座に更新
- **プロジェクト管理資料**: 週次またはマイルストーンごとに更新
- **付録（用語集・参考資料）**: 随時更新

### 差分管理

- ファイル単位での変更履歴を `6_Appendix/change_history.md` に記録
- 大きな変更は別ブランチで実施し、PRでレビュー
- 破壊的変更（既存要件の削除・大幅な変更）は `decisions_log.md` に記録

### レビュープロセス

1. 作成者: ドキュメント作成・更新
2. オーナー: 技術的妥当性のレビュー
3. ステークホルダー: ビジネス要件の確認
4. マージ: 全レビュー完了後
